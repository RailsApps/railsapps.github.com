h1. Example Gemfiles for Rails 3.2 h4. by Daniel Kehoe _Last updated 21 February 2013_ Here are example Gemfiles for typical applications built with Rails 3.2. See "Installing Rails 3.2":http://railsapps.github.io/installing-rails.html for detailed instructions and advice. John Mcdowall offers some useful advice about "Gemfile Best Practices":http://mcdowall.info/posts/gemfile-best-practices-and-discourse/. h2. Rails 3.2 Gemfile A Rails 3.2.12 application includes the following Gemfile when created with the @rails new@ command: 
source 'https://rubygems.org'

gem 'rails', '3.2.12'
gem 'sqlite3'

# Gems used only for assets and not required
# in production environments by default.
group :assets do
  gem 'sass-rails',   '~> 3.2.3'
  gem 'coffee-rails', '~> 3.2.1'
  gem 'uglifier', '>= 1.0.3'
end

gem 'jquery-rails'
Check for the newest versions of these gems: * "rails":http://rubygems.org/gems/rails * "sqlite3":http://rubygems.org/gems/sqlite3 * "sass-rails":http://rubygems.org/gems/sass-rails * "coffee-rails":http://rubygems.org/gems/coffee-rails * "uglifier":http://rubygems.org/gems/uglifier * "jquery-rails":http://rubygems.org/gems/jquery-rails h2. Common Additions Here are gems that are often added to a new Rails application. h4. Gemfile for Ubuntu Linux Since Rails 3.1, a JavaScript runtime has been needed for development on Linux Ubuntu. It is not needed for Mac OS X or Windows. For development on Linux Ubuntu, it's best to install the "Node.js":http://nodejs.org/ server-side JavaScript environment: @sudo apt-get install nodejs@ and set it in your @$PATH@. If you don't install Node.js, you'll need to add this to the Gemfile for each Rails application you build: 
gem 'therubyracer', '>= 0.9.9'
Check for the newest version of "therubyracer":http://rubygems.org/gems/therubyracer. h4. Gemfile for Heroku Heroku provides low cost, easily configured Rails application hosting. For your convenience, see "Tutorial: Rails 3.2 with Ruby 1.9.3 on Heroku":http://railsapps.github.io/rails-heroku-tutorial.html. Heroku uses PostgreSQL as its default database. If you don't want to install PostgreSQL locally, you can set your Gemfile to use SQLite locally and PostgreSQL in production (see "the article":http://railsapps.github.io/rails-heroku-tutorial.html for pros and cons). You'll likely use the Thin web server for a production app though you may wish to use the default Webrick locally. Run @bundle install --without production@ to update your gems. Here's the Gemfile you'd use: 
source 'https://rubygems.org'

gem 'rails', '3.2.12'

# for deployment on Heroku
gem "heroku"
group :development, :test do
  gem 'sqlite3'
end
group :production do
  gem 'pg'
  gem 'thin'
end

# Gems used only for assets and not required
# in production environments by default.
group :assets do
  gem 'sass-rails',   '~> 3.2.3'
  gem 'coffee-rails', '~> 3.2.1'
  gem 'uglifier', '>= 1.0.3'
end

gem 'jquery-rails'
h4. Gemfile for RSpec and Cucumber "RSpec":https://www.relishapp.com/rspec is a popular alternative to the Test::Unit testing framework. "Cucumber":http://cukes.info/ is used with RSpec for Behaviour Driven Development. Several gems are commonly used with RSpec and Cucumber. "Factory_girl":https://github.com/thoughtbot/factory_girl is a fixtures library used to create objects needed for testing. "Capybara":https://github.com/jnicklas/capybara allows testing of web pages by simulating HTTP requests as if someone was using a browser. "Database_cleaner":https://github.com/bmabey/database_cleaner is used to initialize a database to a clean state during tests. "Launchy":http://www.copiousfreetime.org/projects/launchy/ is used to launch a web browser to display the state of a web page during testing (primarily for debugging tests). Here's a Gemfile configured for RSpec and Cucumber: 
source 'https://rubygems.org'

gem 'rails', '3.2.12'
gem 'sqlite3'

# Gems used only for assets and not required
# in production environments by default.
group :assets do
  gem 'sass-rails',   '~> 3.2.3'
  gem 'coffee-rails', '~> 3.2.1'
  gem 'uglifier', '>= 1.0.3'
end

gem 'jquery-rails'

group :development, :test do
  gem "rspec-rails", ">= 2.8.1"
end

group :test do
  gem "factory_girl_rails", ">= 1.6.0"
  gem "cucumber-rails", ">= 1.2.1"
  gem "capybara", ">= 1.1.2"
  gem "database_cleaner"
  gem "launchy"
end
Check for the newest versions of these gems: * "rspec-rails":http://rubygems.org/gems/rspec-rails * "factory_girl_rails":http://rubygems.org/gems/factory_girl_rails * "cucumber-rails":http://rubygems.org/gems/cucumber-rails * "capybara":http://rubygems.org/gems/capybara * "database_cleaner":http://rubygems.org/gems/database_cleaner * "launchy":http://rubygems.org/gems/launchy h2. Gems for the RailsApps Examples Here are Gemfiles for the example apps in the "Rails Apps":http://railsapps.github.io/ repository. A few of the example apps use the "Mongoid":http://mongoid.org/ gem for access to a MongoDB datastore. * "mongoid":http://rubygems.org/gems/mongoid A few of the example apps use "Devise":https://github.com/plataformatec/devise for authentication. * "devise":http://rubygems.org/gems/devise One of the example apps uses "OmniAuth":https://github.com/intridea/omniauth for authentication. * "omniauth":http://rubygems.org/gems/omniauth The example apps have been tested with the indicated versions. For the RailsApps example apps built with Rails 3.2, use the following gems and specify versions optimistically (using the @>=@ operator). The RailsApps example apps have been tested with this configuration: * Ruby 1.9.3 * RubyGems 1.8.24 * Rails 3.2.12 * Rake 10.0 _Note:_ The RailsApps examples are generated with application templates created by the "Rails Apps Composer Gem":https://github.com/RailsApps/rails_apps_composer. For that reason, groups such as @:development@ or @:test@ are specified inline. You can reformat the Gemfiles to organize groups in an eye-pleasing block style (as presented in the Gemfiles shown above). The functionality is the same. h4.
